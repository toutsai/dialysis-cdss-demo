# 透析中心查房與藥物調整輔助系統進度報告

日期：2026-04-27

## 一、系統目的

本專案目標是建立一套適合透析中心使用的查房與臨床決策輔助系統，協助醫師、護理長與護理師在同一個介面中查看病人床位、主要問題、近期事件、透析醫囑、抽血結果與藥物調整建議。

目前系統仍屬於第一版雛形，主要用於流程驗證與同仁試用，不可直接作為正式臨床醫囑或病歷系統使用。

## 二、目前完成進度

### 1. 前台操作介面

已完成 Streamlit 前台雛形，提供網頁式操作介面。使用者可由床位工作台進入個別病人頁面，並切換不同頁籤查看或編輯資料。

目前已建立的主要頁面包括：

- 查房工作台
- 人員設定
- 本院藥物清單
- 規則設定

病人頁面目前包含：

- 主要問題
- 近期事件
- 透析醫囑
- 抽血 / 藥物建議

### 2. 床位與病人清單

已從現有床位排班表匯入病人資料，建立初步床位工作台。使用者可以依照透析班別、床位與病人資訊進入個別病人頁面。

目前床位與病人基本資料來源為既有 Excel 排班表，後續若院內 HIS 可提供資料，將改為由院內系統串接。

### 3. 權限與角色設計

目前已建立初步角色權限概念：

- 醫師：可編輯透析醫囑、本院藥物清單、規則設定。
- 護理長：可編輯人員設定、主要問題、近期事件。
- 護理師：可編輯主要問題、近期事件。
- 非醫師可查看規則設定，但不能修改。

所有角色原則上都可以查看查房資料，編輯權限依工作內容區分。

### 4. 人員設定

已建立人員設定頁面，可新增或停用人員，並設定角色。

目前角色選項包括：

- 醫師
- 護理長
- 護理師

人員異動時，資料會保留，不直接刪除，以利後續追蹤。

### 5. 本院藥物清單

已建立本院藥物清單頁面，供醫師維護與查看常用藥物。

目前藥物類型包括：

- ESA
- 降磷藥
- 降鉀藥
- 副甲狀腺亢進藥物

後續可依本院實際品項調整。

### 6. 規則引擎與建議雛形

已建立初步規則引擎，未來可根據抽血結果與藥物紀錄產生藥物調整建議。

目前規則引擎定位為 deterministic rule engine，也就是先由明確規則產生建議，再視需要讓 Claude 或其他大型語言模型協助整理文字。AI 不直接取代臨床判斷。

目前畫面中的部分抽血與藥物資料仍為測試資料，不可用於臨床決策。

## 三、目前架構決策

經討論後，目前建議系統先採用以下架構：

```text
Streamlit 前台
   ↓
SQLite 單一資料庫檔
   ↓
備份 / 稽核紀錄 / 院內資料串接腳本
```

原先曾評估使用 NocoDB 作為資料管理後台，但因為目前主要操作功能已逐步移至前台，NocoDB 暫時不作為核心依賴。

目前 NocoDB 的定位調整為：

```text
可選資料表管理工具
```

也就是說，未來若需要類似 Excel 或 Airtable 的批次資料管理介面，可以再加入 NocoDB；但系統主流程會以 Streamlit 前台與 SQLite 資料庫為主。

## 四、後續資料串接規劃

後續若資訊室或院內系統可提供資料來源，預計串接以下資料：

- 病人基本資料
- 透析排班與床位
- 抽血報告
- 藥物開立紀錄
- 急診、住院、門診紀錄
- 檢查與手術紀錄

串接原則：

- 醫院系統為正式資料來源。
- 本系統儲存本地快取，供查房與規則引擎使用。
- 若院內資料抓取失敗，系統保留上次成功資料並顯示提示。
- 人工補登資料與院內串接資料需區分來源。

## 五、隱私與資安原則

本系統後續若進入試行，需遵守以下原則：

- 病人資料應留在院內主機或院方核可環境。
- 病歷號與去識別化 ID 應分開管理。
- 若使用 Claude 或其他外部 API，送出的資料必須先去識別化。
- 所有 API 呼叫與重要資料修改應留下稽核紀錄。
- 測試階段應優先使用測試資料或去識別化資料。
- 不應將可識別病人資料放到未核可的外部雲端服務。

## 六、近期待完成工作

### 第一階段：修穩前台

- 確認主要問題可正常新增、修改與儲存。
- 確認近期事件日期、類型、標題與內容欄位可正常使用。
- 確認透析醫囑可依角色權限編輯。
- 確認不同角色的可編輯範圍符合預期。

### 第二階段：整理資料庫

- 以 SQLite 作為主要資料庫。
- 保留 CSV 匯入與匯出能力。
- 整理主要資料表，包括病人、排班、問題列表、事件、透析醫囑、抽血、藥物、建議、人員、藥物清單與稽核紀錄。

### 第三階段：準備同仁試用

- 使用測試資料或去識別化資料。
- 邀請少數同仁試用操作流程。
- 收集使用回饋，例如欄位是否足夠、頁面是否直覺、查房流程是否順。
- 若同仁不在同一個院內網路，可考慮用 Tailscale 做小規模安全測試。

### 第四階段：院內主機與正式試行

- 將系統放到院內主機。
- 設定固定網址或院內連線方式。
- 建立資料庫備份流程。
- 建立操作紀錄與權限管理。
- 規劃 HIS / LIS / EMR 串接。

## 七、目前需要同仁協助回饋的重點

希望同仁試看時可以協助確認：

- 床位工作台是否符合實際查房流程。
- 主要問題欄位是否足夠。
- 近期事件分類是否合適。
- 透析醫囑頁面是否符合醫師需要。
- 藥物清單分類是否符合本院實際用藥。
- 角色權限是否符合臨床工作分工。
- 哪些資料最需要優先從院內系統串接。

## 八、目前結論

目前系統已完成第一版可操作雛形，下一步重點不是立刻擴大功能，而是先把前台操作、角色權限、資料儲存與測試流程修穩。

建議短期方向為：

```text
Streamlit 前台 + SQLite 資料庫 + 去識別化測試資料 + 少數同仁試用
```

待流程確認後，再進一步規劃院內主機部署、正式資料串接、備份機制與稽核紀錄。
